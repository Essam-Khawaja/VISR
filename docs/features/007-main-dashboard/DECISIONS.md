# Feature Decisions: Main Dashboard

## Decision Log

### 2026-05-23 — Dashboard owns UI; 002 and 003 own logic

**Decision:**  
Keep [002](../002-degree-roadmap-engine/) (generation/API) and [003](../003-graph-visualization/) (graph mapping/components) as separate modules. [007](PRD.md) is the composition layer and home route.

**Reason:**  
Separation of concerns: testable roadmap generator and reusable graph without coupling to layout cards.

**Alternatives Considered:**  
- Merge 002+003 into 007 folder only  
- Separate `/graph` as primary route

**Consequence:**  
No duplicate PRDs for generation logic; 007 sub-docs describe dashboard integration only.

---

### 2026-05-23 — React Flow graph for MVP (not Three.js)

**Decision:**  
Hero graph uses React Flow per 003, not custom Three.js radial tree from global architecture PRD.

**Reason:**  
Faster hackathon delivery; 007 still treats graph as hero (60% width). Three.js is post-MVP upgrade.

**Alternatives Considered:**  
- Build Three.js goal tree first

**Consequence:**  
Dashboard DECISIONS override global PRD visualization for MVP scope only.

---

### 2026-05-23 — Rule-based bottleneck and cut list

**Decision:**  
Bottleneck and cut list generated by deterministic rules over profile, roadmap, tasks — not full LLM on every page load.

**Reason:**  
Explainable, demo-stable, no API cost on each dashboard view. Optional LLM polish on text later.

**Alternatives Considered:**  
- Single Claude call returning full StrategyPlan  
- Manual user-set bottleneck

**Consequence:**  
`lib/strategy/*` modules must stay pure functions for unit tests.

---

### 2026-05-23 — Weekly tasks live on separate route

**Decision:**  
Dashboard links to `/tasks` (004) rather than embedding full weekly list on same page.

**Reason:**  
Avoid cluttered hero layout; dashboard = orient + graph + strategy; tasks = execute.

**Alternatives Considered:**  
- Split dashboard 50/50 graph and tasks

**Consequence:**  
Bottleneck card CTA navigates to weekly tasks route.
